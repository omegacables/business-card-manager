# BusinessCard QA テスト観点・チェックリスト

- 作成日: 2026-04-21
- 対象: 本番リリース前の総合QA
- 実行者: 社長（まず自己テスト）→ 必要に応じて外部テスター
- 所要時間: 全項目で **約6〜8時間**

---

## 🎯 本書の使い方

本番リリース前の**最終動作確認**用。以下の順で実施:

1. **Critical テスト**（必須・3時間） — ここで落ちたらリリース延期
2. **Major テスト**（推奨・2時間） — 主要フロー
3. **Minor テスト**（任意・2〜3時間） — 細部の磨き込み

テスト環境:
- **Prod相当のステージング環境**（Vercel Preview推奨）
- 複数ブラウザ: **Chrome / Safari / Firefox**（Safari特に重点）
- デバイス: **PC / iPhone / Android** 最低各1台

# 🔴 Part 1: Critical テスト（本番前必須）

## A. 認証・セキュリティ

### A-1. Auth0 新規登録
- [ ] メールアドレスでの新規登録が動作する
- [ ] Googleログインが動作する
- [ ] LINEログインが動作する（LINE Auth0連携）
- [ ] 登録後、`profiles` テーブルに自動レコード作成されている
- [ ] `subscriptions` テーブルに `plan=free / status=active` が自動作成される
- [ ] 初回ログイン後、ダッシュボードに正常遷移

### A-2. Auth0 ログイン
- [ ] 既存ユーザーのログインが動作
- [ ] 間違ったパスワードで適切なエラー
- [ ] ログアウトが完全に動作（セッション消去）
- [ ] セッション切れで自動的にログイン画面へリダイレクト

### A-3. Safari互換性（特に重点）
- [ ] **Safari（macOS）** でログイン→ダッシュボード遷移
- [ ] **Safari（iPhone）** でログイン→ダッシュボード遷移
- [ ] Cookieが消えずに保持される
- [ ] Private Browsingでもログイン可能

### A-4. データ分離（最重要）
**テストシナリオ**: 2ユーザーA・Bでログインして他方のデータが見えないか確認

- [ ] **ユーザーA** が名刺5枚登録 → ログアウト
- [ ] **ユーザーB** でログイン → **ユーザーAの名刺が見えないこと**
- [ ] **ユーザーB** が名刺3枚登録 → ログアウト
- [ ] **ユーザーA** で再ログイン → **自分の5枚のみ表示、Bの3枚は見えない**
- [ ] **ユーザーB** のURLを**ユーザーA**のブラウザで直叩き
  - 例: `/cards/{userB-card-id}/edit` → **404になるべき**
- [ ] `/api/debug` ルート削除済みか確認 ← **🔴必ず削除**

### A-5. 名刺画像のセキュリティ
- [ ] Storage バケットが **privateに設定**されているか
- [ ] 画像URLを**認証なしブラウザで開く** → アクセス不可になるべき
- [ ] Signed URLの有効期限が設定されているか

## B. OCR・名刺登録機能

### B-1. 画像アップロード
- [ ] JPEG画像のアップロード成功
- [ ] PNG画像のアップロード成功
- [ ] WebP画像のアップロード成功
- [ ] **巨大ファイル（10MB超）**を拒否（上限設定あるか要確認）
- [ ] **非画像ファイル（.exe / .pdf）**を拒否
- [ ] スマホで撮影した縦向き画像が正しい向きで処理される（EXIF Orientation）

### B-2. OCR精度テスト
以下の3〜5種類の名刺で精度確認:
- [ ] **日本語メインの名刺**（会社名・部署・役職・氏名・電話）
- [ ] **英語名刺**
- [ ] **手書き風フォント**の名刺
- [ ] **ロゴ画像あり**の名刺
- [ ] **多言語混在**（日英混在）

各項目の抽出精度:
- [ ] 氏名: 95%以上
- [ ] 会社名: 90%以上
- [ ] メール: 100%（フォーマット明確なので）
- [ ] 電話: 95%以上
- [ ] 部署・役職: 80%以上

### B-3. 月次利用回数制限
- [ ] **Freeプラン**で10枚登録 → 11枚目で上限エラー表示
- [ ] 上限エラー時の**アップグレード誘導**が動作
- [ ] **Proプラン**では無制限に登録可能
- [ ] **月が変わったら**カウントがリセットされる（翌月テスト推奨）
- [ ] **並列リクエスト**で上限を超えないか（セキュリティ）

## C. Stripe決済

### C-1. Checkout（新規購入）
- [ ] Freeプラン → Proプランアップグレードボタンが表示
- [ ] Stripe Checkoutページに遷移
- [ ] テストカード `4242 4242 4242 4242` で決済成功

- [ ] ダッシュボードに「Proユーザー」表示

### C-2. Webhook
- [ ] `checkout.session.completed` を正しく処理
- [ ] `customer.subscription.updated` を正しく処理
- [ ] `customer.subscription.deleted` でFreeに戻る
- [ ] `invoice.payment_failed` で `status=past_due` になる
- [ ] 同じWebhookが**2度来ても冪等**（二重処理なし）

### C-3. Customer Portal
- [ ] 「支払い方法変更」ボタンからStripe Portal遷移
- [ ] カード情報の変更ができる
- [ ] サブスクリプション解約ができる
- [ ] 解約後、期末まではPro継続、期末でFree化

### C-4. エラーパターン
- [ ] **決済失敗カード** `4000 0000 0000 0002` で適切なエラー表示
- [ ] **3Dセキュア認証必要カード** `4000 0025 0000 3155` が動作
- [ ] 決済途中のキャンセルでプラン変更されない

# 🎯 リリース判定基準

### 🟢 Go条件
- **Part 1 Critical 全て合格**
- Part 2 Major の **80%以上** 合格
- **本番環境変数** 全て確認
- **バックアップ** 有効

### 🟡 Hold条件（延期）
- Critical で1件でも失敗
- データ分離テスト（A-4）に懸念
- OCR精度が著しく低い
- Stripe決済が不安定

### 🔴 No-Go条件（中止・根本対応）
- `/api/debug` が残っている
- ストレージバケットがpublic
- LINE検索にinjection脆弱性
- **他ユーザーのデータが1件でも見える**
